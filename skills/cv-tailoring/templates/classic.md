# CV Template: Classic

A single-column, ATS-friendly professional CV template. Clean, minimalist design with clear section hierarchy. Suitable for software engineers, product managers, data scientists, and any professional role where readability and keyword density matter.

---

## Template Identity

| Property | Value |
|---|---|
| Name | Classic |
| Layout | Single-column |
| Target length | 1–2 pages |
| Tone | Professional, direct, achievement-oriented |
| ATS compatibility | Full — no tables, images, or decorative elements |

---

## Section Order

Use this exact order. Optional sections are included only when the profile contains relevant content.

1. **Header** *(always)*
2. **Professional Summary** *(always)*
3. **Key Skills** *(always)*
4. **Technical Skills** *(always for technical roles; omit for non-technical roles)*
5. **Professional Experience** *(always)*
6. **Education** *(always)*
7. **Publications & Open Source** *(optional — only if profile contains patents, papers, or notable open-source work)*
8. **Certifications** *(optional — only if profile contains certifications)*
9. **Awards** *(optional — only if profile contains awards)*
10. **Community & Speaking** *(optional — only if profile contains speaking engagements or volunteering)*

---

## Section Formatting Rules

### 1. Header

```
[Full Name]
[City, Country]  ·  [email]  ·  [linkedin.com/in/handle]  ·  [github.com/handle] (if applicable)
[Language 1 (proficiency)]  ·  [Language 2 (proficiency)]  (omit if only one language or not relevant)
```

- Name on its own line, no title or suffix unless legally part of the name
- Contact details on a single line, separated by ` · ` (space–middot–space)
- Languages on a separate line only when bilingual/multilingual is relevant to the role
- No photo, no date of birth, no nationality (unless legally required)

### 2. Professional Summary

```
[Role identity] with [X]+ years of experience [core domain]. [Achievement 1 with metric]. [Achievement 2 with metric or scope]. [What you bring to this specific role or company].
```

Rules:
- 3–4 lines of continuous prose — **no bullets**
- Opens with role identity + years of experience + core domain
- Contains at least 2 concrete achievements with numbers or scope indicators
- Ends with what the candidate brings to this specific role or team
- Written in third-person implied (no "I") or first-person — be consistent throughout
- **Do not** use generic phrases: "team player", "passionate about", "results-driven" without evidence

### 3. Key Skills

```
Skill 1, Skill 2, Skill 3, Skill 4, Skill 5, Skill 6, Skill 7, Skill 8
```

Rules:
- Single line or compact comma-separated paragraph — **no bullets, no headings within this section**
- 6–12 items maximum
- Mix of business/domain skills and interpersonal skills aligned to the job description
- Do not duplicate items already in Technical Skills

### 4. Technical Skills

```
**Category Label:** Item 1, Item 2, Item 3
**Category Label:** Item 1, Item 2
```

Rules:
- One line per category; category name in **bold** followed by a colon and the items
- Order categories by relevance to the target job (most relevant first)
- Standard category labels: Languages · Frameworks · Cloud & DevOps · Databases · Platforms & Tools · Methodologies
- Include **Methodologies** only when the job description explicitly mentions methodology requirements (e.g., "Scrum experience required")
- 3–8 items per category

### 5. Professional Experience

For each role:
```
**[Job Title]**
[Company]  |  [City, Country]  |  [Start Month Year] – [End Month Year or Present]

- [Action verb] + what you did + how you did it + measurable impact
- [Action verb] + what you did + how you did it + measurable impact
```

Rules:
- Roles in **reverse-chronological order** (most recent first)
- Job title in **bold**
- Company line: Company · Location · Date Range, separated by ` | `
- 3–7 bullets per role, ordered by relevance to the target job (most relevant first)
- **Every bullet** must start with a strong past-tense action verb (present tense for current role)
- **Every bullet** must state impact: metric, scope indicator (team size, user count, system count), or downstream effect
- Reduce bullet count for older or less relevant roles (3 bullets minimum)

Action verb bank:
- Leadership: Led, Directed, Spearheaded, Championed, Founded, Orchestrated, Mentored
- Technical: Built, Designed, Engineered, Implemented, Developed, Architected, Migrated, Refactored
- Impact: Delivered, Achieved, Reduced, Increased, Accelerated, Improved, Eliminated
- Innovation: Pioneered, Initiated, Introduced, Established, Launched, Automated

Bullet quality checklist (apply to every bullet):
- [ ] Starts with a strong action verb
- [ ] States what was done (not just responsibility)
- [ ] States how it was done (technology, method, or approach)
- [ ] States measurable impact or scope (number, %, time, cost, team size, user count)
- [ ] Is a single concise sentence — no paragraph-length bullets

Anti-pattern — duties without outcomes:

| ✗ Weak | ✓ Strong |
|---|---|
| Managed a team of engineers | Led a team of 5 engineers to deliver a payment integration on schedule, reducing checkout abandonment by 12% |
| Responsible for onboarding new hires | Designed onboarding programme for 8 new joiners, cutting time-to-productivity from 6 weeks to 3 |
| Worked on data pipeline reliability | Refactored ETL pipeline, reducing failure rate from 15% to <1% and eliminating weekly manual intervention |

### 6. Education

For each entry:
```
**[Degree and Field]**
[Institution]  |  [Start Year] – [End Year]
[Optional: Thesis: topic | Grade: distinction | Notable achievement]
```

Rules:
- Degrees in reverse-chronological order
- Degree in **bold**
- Institution and date range on the same line, separated by ` | `
- Optional detail line: thesis topic, relevant specialisation, notable grade (only if impressive)
- Do not list every module or course

### 7. Publications & Open Source *(optional)*

```
- [Project or paper title] — [brief description and impact]
- [Patent title or number] — [status: filed/granted] ([Year])
```

Rules:
- Include only if the profile contains verifiable evidence (paper DOI, GitHub repo, patent number)
- Filed patents are weaker signal — include only if directly relevant to the target role
- Open-source contributions: name the project, your specific contribution, and measurable adoption or impact
- Academic publications: name the venue (journal or conference) and year

### 8. Certifications *(optional)*

```
**[Certification Name]** — [Issuing Organisation] ([Month Year])
```

Rules:
- One line per certification; name in **bold**
- Include only current/valid certifications; note expiry if relevant
- Order by recency (most recent first)

### 9. Awards *(optional)*

```
**[Award Name]** — [Context or description] ([Year])
```

Rules:
- Same format as certifications
- Include only externally recognised awards (internal "employee of the month" is weak unless the role specifically asks for leadership recognition)

### 10. Community & Speaking *(optional)*

```
- [Talk title or community role] — [Venue or organisation] ([Year or date range])
```

Rules:
- Bulleted list
- Include speaking engagements, mentoring programmes, open-source maintainership, teaching, or committee roles
- One line per item with enough context to be verifiable

## Content Quality Standards

### Metrics first
Always prefer quantified impact. When exact numbers are unavailable, use scope indicators:
- Counts: "3 product teams", "12 microservices", "50+ AWS resources"
- Volumes: "2M+ daily transactions", "10K+ monthly active users"
- Time: "reducing deployment time from 45 minutes to 8 minutes"
- Cost/revenue: "cutting infrastructure costs by 25%", "€5M+ in monthly transactions"
- Improvement: "from 30% to 85% test coverage", "from 15% to <1% failure rate"

### ATS keyword placement
- Mirror exact phrasing from the job description where truthful
- Include both the acronym and full form on first mention: "Kubernetes (K8s)", "CI/CD (Continuous Integration/Continuous Deployment)"
- Place the most important keywords in the Professional Summary and the first 2 bullets of each role
- Do not keyword-stuff — every keyword must appear in a meaningful context

### Employment gaps
If the profile has a gap > 6 months, address it naturally:
- Brief note in the role preceding or following the gap: e.g., "(Career break: upskilling in cloud architecture)"
- Or a short mention in the Professional Summary if the gap is the most recent period
- Never leave an unexplained gap

### Expanding acronyms
Expand on first mention in the Professional Summary; use the short form thereafter.

## Length Guidelines

| Experience level | Target |
|---|---|
| Early career (0–5 years, 2–3 roles) | 1 page |
| Mid-career (5–10 years, 3–5 roles) | 1–2 pages |
| Senior (10+ years, 5+ roles) | 2 pages maximum |

**Never exceed 2 pages.** To fit: reduce older role bullet counts to 3, condense or omit optional sections, tighten bullet phrasing.
